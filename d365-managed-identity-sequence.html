Dynamics 365 Online — Managed Identity 接入外部接口

三种实现方案的完整时序对比

方案 A：Plugin → Azure Function
方案 B：Power Automate Flow
方案 C：Custom API → Azure Function
Plugin / Custom Action  →  Azure Function (Managed Identity)  →  外部 OAuth2.0 接口
同步请求 响应返回 Azure 内部调用 外部调用
⚠️ 为什么不能直接用 MI Plugin 运行在 D365 托管的 Sandbox 进程，无法访问 Azure IMDS 端点，因此本身无法获取 MI Token。
🔌 Plugin Secure Configuration Function URL 和 Function Key 存储在插件注册的 Secure Config 字段，D365 平台加密存储，仅运行时注入。
✅ Managed Identity 在哪里生效 Azure Function 层启用 System-assigned MI，授予 Key Vault Secrets User 角色，免密读取 Client Secret。
Power Automate Flow (Managed Identity)  →  Key Vault  →  外部 OAuth2.0 接口
同步请求 响应返回 Azure 内部调用 外部调用
✅ 原生支持 Managed Identity Power Automate Flow 的 Azure Key Vault 连接器直接支持托管标识认证，在连接配置中选择即可。
⚡ HTTP 连接器获取 Token 使用 Flow 内置的 HTTP 步骤，POST 到外部 OAuth2.0 Token 端点，配合 Key Vault 取出的 Secret。
⚠️ 注意异步特性 Flow 是异步执行的，如果 Plugin 需要同步等待结果，需要结合 Custom API 回调机制设计。
Custom API + Plugin  →  Azure Function (Managed Identity)  →  外部 OAuth2.0 接口
同步请求 响应返回 Azure 内部调用 外部调用
🛠️ Custom API vs Custom Action Custom API 是更现代的方式，支持显式定义请求/响应参数，有 IsFunction 标志，但底层执行仍是 Plugin。
⚠️ 同样无法直接用 MI Plugin 执行环境与 Custom Action 完全一致，依然需要 Azure Function 作为 MI 代理层。
✅ 同步返回是优势 Custom API 可以同步返回 Azure Function 的处理结果，适合需要实时响应的场景。
